\section*{Resumen}
Este documento técnico presenta una guía metodológica detallada para el diseño, configuración e implementación de un entorno de desarrollo de firmware embebido unificado, orientado a microcontroladores de la familia STM32 mediante el uso del editor de código Visual Studio Code en sistemas operativos Windows. El flujo de desarrollo presentado abarca desde la instalación inicial de las herramientas indispensables de software (Visual Studio Code, STM32CubeMX y CMake), hasta la integración práctica del diseño del hardware utilizando como plataforma física la tarjeta de desarrollo NUCLEO-F303RE.\\

Asimismo, se exponen las mejores prácticas de optimización y organización dentro de VS Code. Esto incluye la creación y activación de perfiles dedicados (Profiles) para el desarrollo con STM32, lo cual previene la saturación de recursos en el IDE, así como la asignación manual del ejecutable de CMake en las configuraciones globales. A través del uso de la extensión oficial de STMicroelectronics, se detalla la vinculación de la toolchain GCC y el depurador físico ST-Link a través de un GDB Server local para realizar la carga y monitoreo del firmware.

\section{Aplicaciones requeridas para la creación de proyectos STM32}

Para poder inicializar, configurar y desarrollar un nuevo proyecto destinado a tarjetas de desarrollo o microcontroladores STM en Visual Studio Code, es fundamental descargar e instalar las siguientes herramientas en su versión para Windows:
\begin{description}
	\item[Visual Studio Code para Windows:] El entorno de desarrollo (IDE) ligero donde se escribirá, compilará y depurará el código C/C++. Se recomienda instalar posteriormente las extensiones oficiales de STMicroelectronics (como \textit{STM32 VS Code Extension}).
	\begin{itemize}
		\item Enlace de descarga: \href{https://code.visualstudio.com/}{code.visualstudio.com}
	\end{itemize}
	
	\item[STM32CubeMX:] Herramienta gráfica oficial de STMicroelectronics que se utiliza para configurar los periféricos del microcontrolador (pines GPIO, relojes del sistema, puertos USART, etc.) y generar el código de inicialización base del proyecto.
	\begin{itemize}
		\item Enlace de descarga: \href{https://www.st.com/en/development-tools/stm32cubemx.html}{st.com/stm32cubemx}
	\end{itemize}
	
	\item[CMake:] El sistema de generación de compilación multiplataforma. Las nuevas plantillas de proyectos de STM32CubeMX utilizan CMake para gestionar y organizar la compilación de todos los archivos de código fuente.
	\begin{itemize}
		\item Enlace de descarga: \href{https://cmake.org/download/}{cmake.org}
	\end{itemize}
\end{description}
\begin{InfoBox}{Variable de Entorno (PATH)}
	Durante la instalación de \textbf{CMake}, asegúrese de marcar la opción que dice \textbf{"Add CMake to the system PATH for all users"} (o para el usuario actual). Esto es indispensable para que Visual Studio Code pueda localizar y ejecutar CMake de forma automática desde la terminal.
\end{InfoBox}

\subsection{Extensiones Recomendadas para Visual Studio Code}
Una vez instalado Visual Studio Code, es necesario equiparlo con las extensiones adecuadas para habilitar el soporte de desarrollo y depuración de microcontroladores. Abra el panel de Extensiones en VS Code mediante el atajo \keys{Ctrl + Shift + X} e instale las siguientes herramientas:
\begin{itemize}
	\item \textbf{C/C++ Extension Pack:} Proporciona soporte esencial para el lenguaje, incluyendo el autocompletado inteligente (IntelliSense), formateo automático, navegación por símbolos y depuración básica.
	\item \textbf{CMake Tools:} Permite a VS Code interpretar, configurar y compilar proyectos que utilicen sistemas de archivos basados en CMake de forma gráfica e integrada.
	\item \textbf{Serial Monitor:} Extensión oficial de Microsoft que integra un monitor de terminal serie directamente en las pestañas del editor para interactuar con los datos enviados por la tarjeta STM32.
	\item \textbf{STM32CubeIDE for Visual Studio Code:} La extensión principal de STMicroelectronics. Al instalar esta extensión, se instalará de forma automática el paquete completo que incluye 15 extensiones integradas, entre las que destacan:
	\begin{itemize}
		\item \textit{STM32Cube Core} (Núcleo de soporte básico)
		\item \textit{STM32Cube Bundles Manager} (Gestor de recursos y librerías)
		\item \textit{STM32Cube CMake Support} (Integración y soporte para compilación basada en CMake)
		\item \textit{STM32Cube Project Manager} (Importación y creación de proyectos desde CubeMX)
		\item \textit{STM32Cube Build Analyzer} (Analizador visual del resultado de compilación y uso de memoria)
		\item \textit{STM32Cube Debug Core} (Motor de depuración de firmware)
		\item \textit{STM32Cube Debug Generic GDB Server} (Servidor GDB para depuración genérica)
		\item \textit{STM32Cube Debug STLink GDB Server} (Servidor GDB específico para depuradores físicos ST-Link de ST)
		\item \textit{STM32Cube Debug Jlink GDB Server} (Servidor GDB específico para depuradores Segger J-Link)
		\item \textit{STM32Cube RTOS Views} (Vistas de depuración en tiempo real para sistemas operativos RTOS)
		\item \textit{STM32Cube Registers View} (Visor detallado del estado de los registros SFR del microcontrolador)
		\item \textit{STM32Cube clangd} (Servicio de lenguaje avanzado basado en Clang para análisis y refactorización)
		\item \textit{Memory Inspector} (Herramienta gráfica para evaluar las secciones de RAM y Flash del chip)
		\item \textit{Web and Desktop Serial Monitor} (Monitor serie alternativo y avanzado para tramas de comunicación)
	\end{itemize}
\end{itemize}
\begin{InfoBox}{Instalación del Paquete de STMicroelectronics}
	Al instalar la extensión \textbf{STM32CubeIDE for Visual Studio Code}, VS Code descargará e instalará automáticamente en segundo plano todas las demás extensiones asociadas de STMicroelectronics. No es necesario buscarlas ni instalarlas individualmente.
\end{InfoBox}

Una vez finalizada la instalación de estas herramientas, observará que en la \textbf{barra de actividad lateral izquierda} de Visual Studio Code aparecerán dos nuevos iconos dedicados:
\begin{enumerate}
	\item El icono de \textbf{CMake} (representado por una figura geométrica tridimensional), desde el cual podrá configurar y compilar objetivos.
	\item El logotipo oficial de \textbf{STMicroelectronics}, el cual da acceso directo a la gestión del proyecto, visualización de registros y herramientas de análisis del microcontrolador.
\end{enumerate}

\subsection{Descarga y Configuración de CMake}
Para realizar la descarga e instalación de CMake en su sistema, siga los siguientes pasos:
\begin{enumerate}
	\item Ingrese a la página oficial de descargas de CMake a través del enlace proporcionado anteriormente.
	\item Al cargar el portal, identificará que las descargas se dividen en dos grandes opciones:
	\begin{itemize}
		\item \textbf{Source Distributions:} Bloques que contienen únicamente el código fuente original del programa, recomendados si se desea compilar la herramienta desde cero.
		\item \textbf{Binary Distributions:} Paquetes con ejecutables precompilados y listos para ser utilizados directamente en el sistema operativo.
	\end{itemize}
	\item Desplácese a la sección de \textbf{Binary Distributions} y descargue el paquete comprimido denominado \textbf{\texttt{cmake-4.3.2-windows-x86\_64.zip}} adaptado para la arquitectura de Windows de 64 bits.
\end{enumerate}
\begin{InfoBox}{Uso de la Versión Portátil (ZIP)}
	Al tratarse de una versión comprimida en formato \texttt{.zip}, deberá extraer el contenido en una carpeta permanente dentro de su disco duro (por ejemplo, en \directory{C:/Program Files/CMake} o \directory{C:/CMake}). No olvide agregar la ruta de la subcarpeta \texttt{bin} a las variables de entorno (PATH) del sistema para que Visual Studio Code pueda ejecutar CMake de forma global.
\end{InfoBox}

\section{Configuraciones Previas en Visual Studio Code}
Antes de proceder con la inicialización y creación de cualquier proyecto, es muy importante realizar un par de ajustes de preparación dentro de Visual Studio Code. Estas configuraciones se realizan una sola vez en el editor, por lo que no será necesario repetirlas cada vez que cree un nuevo proyecto en el futuro.
\subsection{Creación y Uso de Perfiles (Profiles)}
Los \textbf{Perfiles (Profiles)} en Visual Studio Code son una característica que permite guardar y aislar conjuntos específicos de configuraciones, temas, atajos de teclado, estados de la interfaz y extensiones instaladas.
\subsubsection{¿Para qué sirven y cuáles son sus ventajas?}
El desarrollo embebido para microcontroladores STM32 requiere herramientas y dependencias muy particulares (compiladores, extensiones de depuración, etc.). El uso de un perfil dedicado aporta las siguientes ventajas:
\begin{itemize}
	\item \textbf{Aislamiento y Organización:} Mantiene las extensiones de STM32 separadas de las extensiones que utilice para otros tipos de programación (como desarrollo web, Python, etc.), evitando conflictos y saturación de menús.
	\item \textbf{Optimización del Rendimiento:} Al cargar únicamente las extensiones específicas de STM32 en este perfil, VS Code consume menos memoria RAM y el editor arranca mucho más rápido.
	\item \textbf{Portabilidad y Respaldo:} Permite exportar toda la configuración de desarrollo en un solo archivo e importarlo en cualquier otra máquina para tener el entorno de trabajo listo al instante.
\end{itemize}
\subsubsection{Pasos para crear un nuevo perfil}
Para configurar un perfil exclusivo para su trabajo con tarjetas STM, siga los siguientes pasos:
\begin{enumerate}
	\item Diríjase al extremo inferior de la barra de actividad lateral izquierda y haga clic en el icono del \textbf{engranaje} (Administración).
	\item En el menú contextual que aparece, seleccione la opción \textbf{Profiles} $\rightarrow$ \textbf{Profiles} (o \textbf{Create Profile...} según la versión de su editor).
	\item Esto abrirá una pestaña de configuración llamada \textbf{"Profiles"}. En el campo de texto etiquetado como \textbf{Name}, asigne un nombre representativo para este perfil, por ejemplo: \texttt{STM32} (tal como se ilustra en la figura \ref{fig_stm32_10}).
	
	\begin{figure}[H]
		\centering
		\includegraphics[width=0.85\textwidth]{stm32_10}
		\caption{Creación del perfil dedicado "STM32" en Visual Studio Code.}\label{fig_stm32_10}
	\end{figure}
	
	\item Una vez ingresado el nombre, puede proceder a cerrar la pestaña de perfiles; el entorno guardará los cambios automáticamente.
\end{enumerate}
\subsubsection{Activación del perfil}
Para activar y comenzar a trabajar bajo este entorno, haga clic nuevamente en el icono del \textbf{engranaje} en la esquina inferior izquierda, desplace el cursor sobre la opción \textbf{Profiles} y haga clic en el perfil previamente creado (\textbf{STM32}) en la lista desplegable. Esto cambiará todo el contexto del editor a tu espacio de trabajo configurado.

\subsection{Configuración de la Ruta del Ejecutable de CMake}
De acuerdo con la documentación y advertencias provistas por la extensión oficial \textbf{CMake Tools}, para que la automatización de compilación opere sin inconvenientes es necesario cumplir con el siguiente requerimiento:
\begin{quote}
	\textit{"Ensure that either you've added your CMake executable to your PATH, or you've adjusted the cmake.cmakePath setting to point to your CMake executable."}
\end{quote}
Para solventar esta directiva indicando explícitamente a VS Code dónde se encuentra el ejecutable de CMake en su computadora, realice el siguiente procedimiento:
\begin{enumerate}
	\item Diríjase al icono del \textbf{engranaje} (Administración) en la barra de actividad y elija la opción \textbf{Settings} (o use el atajo rápido \keys{Ctrl + ,}).
	\item En la barra de búsqueda de la pestaña de configuraciones que se despliega, escriba la frase de búsqueda: \texttt{path cmake}.
	\item Identifique el campo de configuración titulado \textbf{"Cmake: Cmake Path"} (cuya etiqueta de descripción indica: \textit{Name/path of the CMake executable to use}).
	\item En la caja de texto de esta opción, introduzca la ruta exacta en su disco duro donde se localiza el binario de CMake. Para esta instalación, la ruta corresponde a: \\
	\texttt{C:\textbackslash Program Files\textbackslash CMake\textbackslash bin\textbackslash cmake} (tal como se ilustra en la figura \ref{fig_stm32_11}).
	
	\begin{figure}[H]
		\centering
		\includegraphics[width=0.9\textwidth]{stm32_11}
		\caption{Asignación de la ruta de CMake en el campo Cmake Path de VS Code.}\label{fig_stm32_11}
	\end{figure}
	
	\item Cierre la ventana de configuración; Visual Studio Code registrará y salvará las modificaciones de forma automática e inmediata.
\end{enumerate}

\subsection{Configuración de la Ruta del Ejecutable de STM32CubeMX}
Para garantizar que Visual Studio Code pueda lanzar la herramienta gráfica de configuración física de pines sin inconvenientes, es altamente recomendable configurar de forma explícita la ruta de instalación de STM32CubeMX. Esto previene un error común en el que el editor muestra una notificación indicando que no se ha encontrado la ruta para abrir la aplicación.
Para configurar esta ruta, realice el siguiente procedimiento:
\begin{enumerate}
	\item Abra la ventana de configuraciones globales de Visual Studio Code mediante el atajo de teclado \keys{Ctrl + ,} (o dirigiéndose al icono del engranaje en la esquina inferior izquierda y seleccionando \textbf{Settings}).
	
	\item En la barra de búsqueda de la pestaña de configuraciones, escriba el texto de búsqueda: \texttt{path stm32cubemx}.
	
	\item Identifique la sección de configuración correspondiente a la extensión de STMicroelectronics, específicamente el campo titulado: \\
	\textbf{Stm32cube-ide-core $\rightarrow$ Configuration $\rightarrow$ Product STM32 Cube MX: Executable Path} \\
	\textit{(Absolute path to the STM32CubeMX executable)}.
	
	\item Introduzca la ruta absoluta exacta donde se encuentra instalado el ejecutable en su sistema. Para esta versión, la ruta corresponde a: \\
	\texttt{C:\textbackslash Users\textbackslash casti\textbackslash AppData\textbackslash Local\textbackslash Programs\textbackslash STM32CubeMX\_6.17.0\textbackslash STM32CubeMX.exe} \\
	(tal como se ilustra en la figura \ref{fig_stm32_18}).
	
	\begin{figure}[H]
		\centering
		\includegraphics[width=0.9\textwidth]{stm32_18}
		\caption{Asignación de la ruta absoluta del ejecutable de STM32CubeMX en VS Code.}\label{fig_stm32_18}
	\end{figure}
	
	\item Cierre la ventana de configuración; Visual Studio Code registrará y salvará las modificaciones de forma automática e inmediata.
\end{enumerate}
\begin{WarningBox}{Error de Lanzamiento de STM32CubeMX}
	Si omite esta configuración, al hacer clic en las opciones del panel lateral de ST como \textbf{Launch STM32CubeMX}, es posible que VS Code intente buscar el ejecutable en las rutas por defecto del sistema y falle si se trata de una instalación local de usuario, arrojando una alerta de error. Configurar la ruta absoluta soluciona este comportamiento de manera permanente.
\end{WarningBox}

\section{Creación de un nuevo proyecto usando STM32CubeMX}
Una vez que el entorno de desarrollo y las extensiones se han instalado correctamente, puede proceder con la creación y configuración inicial de su firmware. El flujo de trabajo oficial de ST en Visual Studio Code le permite gestionar todo el proceso desde el panel integrado:
\begin{enumerate}
	\item Diríjase a la \textbf{barra de actividad lateral izquierda} en Visual Studio Code.
	\item Haga clic en el icono oficial de \textbf{STM32Cube} (el logotipo con las letras \textbf{ST}).
	\item Se abrirá un panel lateral en el editor que mostrará las siguientes 7 opciones de configuración y creación (figura \ref{fig1}):
	\begin{itemize}
		\item \textbf{Launch STM32CubeMX:} Abre de forma externa la herramienta gráfica oficial de ST para configurar el hardware y los periféricos del microcontrolador.
		\item \textbf{Launch STM32CubeMX2:} Método secundario de acceso rápido para arrancar la interfaz gráfica de CubeMX.

\subsection{El problema de la impresión de Punto Flotante}
Una vez iniciado el monitoreo del puerto, observará que la consola imprime las cadenas, pero las variables decimales de tipo \texttt{double} o \texttt{float} se muestran completamente vacías en las llamadas de \texttt{sprintf}:

\begin{lstlisting}[language=DOS]
	---- Opened the serial port COM3 ----
	CONFIGURATION Register = 0xCCA4 
	Low Limit Lux =  
	High Limit Lux =  
	Lux =  
\end{lstlisting}

Esto ocurre porque la librería estándar compacta de C para microcontroladores ARM GCC (\textit{newlib-nano}) **deshabilita por defecto el formateo de punto flotante** en funciones como \texttt{printf} y \texttt{sprintf} para ahorrar memoria Flash en el chip.

\subsection{Habilitación de Flotantes en el enlazador de CMake}
Para habilitar el soporte de punto flotante, debemos modificar el archivo **\texttt{CMakeLists.txt}** principal de la raíz del proyecto para indicarle al enlazador del compilador GCC (\textit{linker}) que cargue la biblioteca con soporte para flotantes usando la bandera **\texttt{-u \_printf\_float}**.

Abra su archivo **\texttt{CMakeLists.txt}** y agregue la siguiente directiva de opciones de enlace (\texttt{target\_link\_options}) debajo de la declaración del proyecto:

\begin{lstlisting}[style=CMakeStyle]
	# Habilitar el soporte de punto flotante para printf/sprintf (newlib-nano)
	target_link_options(${CMAKE_PROJECT_NAME} PRIVATE
    -u_printf_float
	-u_scanf_float
	)
\end{lstlisting}

Una vez agregado este comando al archivo, guarde los cambios, reconfigure CMake y compile nuevamente el proyecto. Al iniciar el monitor serie, verá los valores de luz decimales medidos por el sensor \texttt{OPT3001} imprimiéndose correctamente.

\begin{WarningBox}{¿Por qué son necesarias estas banderas?}
Los microcontroladores tienen memoria Flash muy limitada. Por ello, la biblioteca estándar de C utilizada en sistemas embebidos (newlib-nano) viene optimizada de fábrica:
\begin{itemize}
	\item Por defecto, deshabilita el soporte para formatear números con decimales (\texttt{\textbf{float}} y \texttt{\textbf{double}}) en funciones de texto para ahorrar entre 10 KB y 15 KB de memoria Flash.
	\item Si intentas usar \texttt{\textbf{\%f}} en un \texttt{sprintf} o \texttt{\textbf{\%lf}} en un \texttt{sscanf} sin estas banderas, tu programa no podrá imprimir ni leer los decimales (se mostrarán vacíos o la función fallará).
\end{itemize}
\end{WarningBox}

El comportamiento y la función técnica de cada una de estas directivas de enlace se detalla a continuación:
\begin{description}
	\item[\texttt{-u \_printf\_float}:] Fuerza al enlazador (\textit{linker}) a cargar el símbolo y la versión de las funciones de salida de texto (\texttt{printf}, \texttt{sprintf}, \texttt{snprintf}) compatibles con magnitudes decimales. Esto habilita el formateo correcto de especificadores como \texttt{\%f}, \texttt{\%e} y \texttt{\%g}.\\
	Permite que puedas hacer \textbf{\texttt{sprintf(msg, "Voltaje = \%.2f V\textbackslash r\textbackslash n", 3.32)}}; y que se imprima "Voltaje = 3.32 V" en la consola serie en lugar de salir vacío.
	
	\item[\texttt{-u \_scanf\_float}:] Fuerza al enlazador a inyectar el soporte de punto flotante en las funciones de análisis e ingreso de datos (\texttt{scanf}, \texttt{sscanf}). Esto permite que el sistema pueda interpretar y parsear magnitudes decimales representadas en formato de texto a variables reales en memoria (usando especificadores como \texttt{\%f} o \texttt{\%lf}).\\
	 Permite que la línea \textbf{\texttt{sscanf(rxBuffer, "SET:\%d:\%lf", \&max\_i, \&shunt\_r)};} pueda extraer con éxito un valor decimal (como 0.00512 de resistencia shunt) a tu variable double. Si no tuvieras activada esta bandera, sscanf fallaría y nunca actualizaría la calibración del sensor.
\end{description}

\subsection{Comparativa de Métodos para Habilitar Punto Flotante}

Existen dos alternativas viables para indicarle al compilador GCC que incluya el soporte de punto flotante en las funciones de formateo (\texttt{printf}/\texttt{sprintf}). Ambos métodos logran el mismo resultado físico (inyectar la bandera \texttt{-u \_printf\_float} en el linker), pero difieren en el nivel de configuración:

\begin{description}
	\item[Método A: Modificación en la Toolchain (\texttt{gcc-arm-none-eabi.cmake})]~\\
	Consiste en editar el archivo de configuración del compilador ubicado en \directory{cmake/gcc-arm-none-eabi.cmake}, modificando la definición de las banderas de enlace globales de la siguiente manera:
	
	\begin{lstlisting}[style=CMakeStyle]
# Antes:
set(CMAKE_EXE_LINKER_FLAGS "${CMAKE_EXE_LINKER_FLAGS} --specs=nano.specs")
		
# Después:
set(CMAKE_EXE_LINKER_FLAGS "${CMAKE_EXE_LINKER_FLAGS} --specs=nano.specs -u _printf_float")
	\end{lstlisting}
	
	\begin{itemize}
		\item \textbf{Ventajas:} Mantiene el script de construcción principal \texttt{CMakeLists.txt} limpio de directivas de compilación específicas de la arquitectura de hardware.
		\item \textbf{Desventajas:} Si en el futuro realiza cambios de hardware en STM32CubeMX y regenera el proyecto, existe el riesgo de que los archivos de la carpeta \directory{cmake/} se sobreescriban y se pierda la configuración.
	\end{itemize}
	
	\item[Método B: Modificación en el Proyecto (\texttt{CMakeLists.txt})]~\\
	Consiste en agregar la opción de enlace directamente sobre el objetivo ejecutable en el archivo principal de la raíz:
	
	\begin{lstlisting}[style=CMakeStyle]
# Habilitar soporte de flotantes en el proyecto principal
target_link_options(${CMAKE_PROJECT_NAME} PRIVATE -u _printf_float)
	\end{lstlisting}
	
	\begin{itemize}
		\item \textbf{Ventajas:} Sigue los estándares modernos de desarrollo en CMake (\textit{target-centric}), es visible inmediatamente al abrir el proyecto y es completamente persistente ante cualquier regeneración de código de CubeMX.
		\item \textbf{Desventajas:} Mezcla banderas específicas del compilador GCC dentro del archivo de construcción principal.
	\end{itemize}
\end{description}

\begin{InfoBox}{Recomendación Práctica}
	Aunque ambos métodos producen exactamente el mismo binario final, el \textbf{Método B} (modificar el archivo \texttt{CMakeLists.txt}) es el más recomendado para tus proyectos, ya que te garantiza que no perderás la visualización de decimales si necesitas volver a configurar pines o relojes en STM32CubeMX más adelante.
\end{InfoBox}

\section{Gestión de Múltiples Archivos \texttt{main} en un solo Proyecto}

Durante la fase de prototipado o pruebas de un nuevo sensor (como el sensor de humedad y temperatura HDC1080), es común requerir la creación de diferentes scripts de prueba con lógicas e implementaciones distintas. Crear un proyecto nuevo desde cero en STM32CubeMX para cada prueba resulta ineficiente, consume excesivo almacenamiento en el disco duro y entorpece el flujo de trabajo.

Para evitar esto, podemos crear múltiples archivos principales en la misma carpeta del proyecto (por ejemplo: \texttt{main.c}, \texttt{main\_1.c}, \texttt{main\_2.c}). Sin embargo, el estándar de programación en C impone una restricción ineludible: \textbf{un ejecutable compilado solo puede poseer una única función \texttt{int main(void)}}. Si intentamos compilar el proyecto directamente con más de un archivo \texttt{main} activo, el enlazador (\textit{linker}) detendrá el proceso arrojando el error crítico:
\begin{lstlisting}[language=DOS]
multiple definition of `main'
multiple definition of `Error_Handler'
\end{lstlisting}
A continuación, se exponen los diferentes métodos para gestionar esta estructura de forma de ordenar y excluir los archivos de prueba inactivos según el entorno utilizado.

\subsection{Método 1: Exclusión de archivos en STM32CubeIDE}
Si está programando bajo el entorno de desarrollo oficial STM32CubeIDE, el gestor de compilación interna basado en Eclipse permite excluir archivos individuales de la compilación de forma visual e intuitiva:
\begin{enumerate}
	\item Agregue sus archivos secundarios a la carpeta \directory{Core/Src} con nombres descriptivos (ej. \texttt{main\_1.c}).
	\item En el panel del \textbf{Project Explorer} (lado izquierdo), haga clic derecho sobre el archivo que no desea compilar en su sesión de pruebas actual.
	\item Desplácese a la opción \textbf{Resource Configurations} $\rightarrow$ \textbf{Exclude from Build...}
	\item En la ventana que aparece, marque todas las configuraciones de compilación existentes (usualmente \texttt{Debug} y \texttt{Release}) y presione el botón \textbf{OK}.
	\item Observará que el archivo excluido se torna de un color grisáceo y su icono presenta una franja diagonal. El compilador lo omitirá por completo en la siguiente compilación. Para activarlo nuevamente, realice el mismo procedimiento y desmarque las casillas de exclusión.
\end{enumerate}

\subsection{Método 2: Compilación Condicional en VS Code mediante CMake}
En Visual Studio Code, cuando el proyecto utiliza un sistema de construcción guiado por CMake, el proceso de exclusión debe controlarse escribiendo una variable de selección condicional dentro del archivo \textbf{\texttt{CMakeLists.txt}} principal en la raíz del proyecto.

\subsubsection{Paso A: Modificación en \texttt{CMakeLists.txt} principal}
Abra el archivo \texttt{CMakeLists.txt} raíz y modifique la sección que define la creación del archivo ejecutable (\texttt{add\_executable}) implementando la siguiente lógica:

\begin{lstlisting}[style=CMakeStyle]
# 1. Definir una variable de control para seleccionar el main activo
# Opciones validas: "MAIN" (para main.c), "TEST1" (para main_1.c), "TEST2" (para main_2.c)
set(ACTIVE_MAIN "TEST1")

# 2. Asignar de manera condicional la ruta del archivo fisico
if(ACTIVE_MAIN STREQUAL "MAIN")
    set(MAIN_SOURCE "Core/Src/main.c")
    message(STATUS "Compilando: MAIN PRINCIPAL (Core/Src/main.c)")
elseif(ACTIVE_MAIN STREQUAL "TEST1")
    set(MAIN_SOURCE "Core/Src/main_1.c")
    message(STATUS "Compilando: PRUEBA 1 (Core/Src/main_1.c)")
elseif(ACTIVE_MAIN STREQUAL "TEST2")
    set(MAIN_SOURCE "Core/Src/main_2.c")
    message(STATUS "Compilando: PRUEBA 2 (Core/Src/main_2.c)")
else()
    message(FATAL_ERROR "ACTIVE_MAIN invalida. Usa 'MAIN', 'TEST1' o 'TEST2'.")
endif()

# 3. Crear el objeto ejecutable a partir de la fuente dinamica seleccionada
add_executable(${CMAKE_PROJECT_NAME}
    ${MAIN_SOURCE}
)
\end{lstlisting}

\subsubsection{Paso B: Modificación crítica en el script autogenerado de CubeMX}
\begin{WarningBox}{La trampa de stm32cubemx/CMakeLists.txt}
	Aunque modifique el archivo raíz anterior, la compilación de CMake volverá a fallar arrojando errores de duplicidad. Esto se debe a que la herramienta STM32CubeMX genera su propio subarchivo de configuración en \directory{cmake/stm32cubemx/CMakeLists.txt}, en el cual agrega de forma fija e incondicional el archivo \texttt{main.c} a la variable \texttt{MX\_Application\_Src}:
	\begin{lstlisting}[style=CMakeStyle]
set(MX_Application_Src
    ${CMAKE_CURRENT_SOURCE_DIR}/../../Core/Src/main.c
    ${CMAKE_CURRENT_SOURCE_DIR}/../../Core/Src/stm32f3xx_it.c
    ...
	\end{lstlisting}
	Para solucionarlo, debe abrir dicho subarchivo y **comentar o eliminar** la línea correspondiente al archivo principal:
	\begin{lstlisting}[style=CMakeStyle]
set(MX_Application_Src
    # ${CMAKE_CURRENT_SOURCE_DIR}/../../Core/Src/main.c  # Excluido para control condicional
    ${CMAKE_CURRENT_SOURCE_DIR}/../../Core/Src/stm32f3xx_it.c
    ...
	\end{lstlisting}
\end{WarningBox}

Una vez completados estos dos pasos, para cambiar de una prueba a otra solo debe alterar el valor del string en la variable \texttt{ACTIVE\_MAIN} de su \texttt{CMakeLists.txt} raíz, guardar y presionar \textbf{Build} (o \keys{F7}) en VS Code.

\subsection{Método 3: Truco Universal (Directivas del Preprocesador de C)}
Este método representa la alternativa favorita de muchos desarrolladores, ya que es \textbf{completamente independiente del entorno de desarrollo (IDE) o del sistema de construcción} (Make, CMake, Ninja, etc.), y evita tener que realizar modificaciones manuales sobre configuraciones del proyecto o del compilador.

Consiste en crear un único archivo central de cabecera que defina cuál es la prueba activa, e inhabilitar condicionalmente el código de los archivos restantes utilizando el preprocesador de C.

\begin{enumerate}
	\item \textbf{Creación de la cabecera de control:}
	Cree un archivo de configuración en la ruta \texttt{Core/Inc/test\_config.h} con el siguiente contenido:
	\begin{lstlisting}[style=CMakeStyle, caption={Archivo de configuración de pruebas test\_config.h.}]
// Core/Inc/test_config.h
#ifndef TEST_CONFIG_H_
#define TEST_CONFIG_H_

// Descomenta unicamente la prueba que desees ejecutar:
#define RUN_MAIN_PRODUCTION
//#define RUN_TEST_1
//#define RUN_TEST_2

#endif
	\end{lstlisting}
	
	\item \textbf{Protección condicional de los archivos de código fuente:}
	En cada uno de sus archivos principales de pruebas, envuelva todo el contenido del archivo (incluyendo la declaración del `main` y las funciones secundarias) dentro de una directiva condicional \texttt{\#ifdef}:
	
	\begin{itemize}
		\item \textbf{En \texttt{Core/Src/main.c}:}
		\begin{lstlisting}[style=CMakeStyle]
#include "test_config.h"
#ifdef RUN_MAIN_PRODUCTION

int main(void) {
    // Codigo de produccion principal...
}

#endif
		\end{lstlisting}
		
		\item \textbf{En \texttt{Core/Src/main\_test1.c}:}
		\begin{lstlisting}[style=CMakeStyle]
#include "test_config.h"
#ifdef RUN_TEST_1

int main(void) {
    // Codigo para probar el sensor HDC1080...
}

#endif
		\end{lstlisting}
		
		\item \textbf{En \texttt{Core/Src/main\_test2.c}:}
		\begin{lstlisting}[style=CMakeStyle]
#include "test_config.h"
#ifdef RUN_TEST_2

int main(void) {
    // Codigo para otra prueba...
}

#endif
		\end{lstlisting}
	\end{itemize}
\end{enumerate}

\begin{InfoBox}{¿Cómo funciona bajo el capó?}
	El compilador compilará físicamente todos los archivos \texttt{.c} listados en tu proyecto, pero el preprocesador de C evaluará las directivas antes de generar el código de máquina. Al estar activa únicamente una directiva en \texttt{test\_config.h}, los archivos inactivos se procesarán como si estuvieran totalmente vacíos tras el paso del preprocesador. De este modo, solo existirá una única definición de la función \texttt{main()} real en tiempo de enlazado, previniendo el error de colisión.
\end{InfoBox}

\subsection{El problema visual de IntelliSense en los archivos inactivos}
Cuando este esquema se implementa con éxito, es normal experimentar un error de interfaz en Visual Studio Code. Al abrir en el editor los archivos que se encuentran inactivos en ese momento (ej. \texttt{main\_2.c} si la variable de selección es \texttt{TEST1}), el sistema desplegará advertencias de inclusión y líneas onduladas de color rojo en las librerías, acompañadas del siguiente mensaje en los problemas:
\begin{lstlisting}[language=DOS]
#include errors detected. Consider updating your compile_commands.json or includePath.
cannot open source file "main.h"
cannot open source file "HDC1080.h"
\end{lstlisting}

\subsubsection{¿Por qué ocurre?}
Este comportamiento se debe a que la extensión oficial C/C++ de VS Code utiliza la base de datos de compilación denominada \texttt{compile\_commands.json} para conocer las rutas de inclusión del proyecto. Como CMake compila de forma selectiva, \textbf{los archivos inactivos se omiten de la base de datos}. Al no estar listados, IntelliSense no sabe qué rutas de hardware y drivers pertenecen a estos archivos y entra en conflicto, aunque el firmware compile sin problemas.

Para eliminar estas falsas alertas visuales y habilitar el autocompletado en todas las pruebas a la vez, se presentan dos posibles alternativas:

\begin{description}
	\item[Alternativa A: Búsqueda Recursiva de Cabeceras en VS Code]~\\
	Para abrir el archivo de configuración de C/C++ en VS Code, presione la combinación de teclas \keys{Ctrl + Shift + P} para desplegar la paleta de comandos y escriba: \textbf{C/C++: Edit Configurations (JSON)}. Esto abrirá o creará el archivo de configuración en la ruta \texttt{.vscode/c\_cpp\_\\properties.json}.

	Si el archivo no cuenta con el arreglo de rutas de inclusión (lo cual es común en proyectos nuevos), se visualizará de la siguiente manera:
	\begin{lstlisting}[language=json, caption={Archivo c\_cpp\_properties.json original (Sin includePath)}]
{
    "configurations": [
        {
            "name": "STM32",
            "compileCommands": "${workspaceFolder}/build/Debug/compile_commands.json",
            "intelliSenseMode": "windows-msvc-x64",
            "cStandard": "c17",
            "cppStandard": "c++17"
        }
    ],
    "version": 4
}
	\end{lstlisting}

	Para habilitar la indexación recursiva de cabeceras en todos los archivos, se debe modificar agregando el parámetro \texttt{"includePath"} y la regla recursiva (asegúrese de agregar la coma de separación en la propiedad anterior):
	\begin{lstlisting}[language=json, caption={Archivo c\_cpp\_properties.json optimizado (Con includePath)}]
{
    "configurations": [
        {
            "name": "STM32",
            "compileCommands": "${workspaceFolder}/build/Debug/compile_commands.json",
            "includePath": [
                "${workspaceFolder}/**"
            ],
            "intelliSenseMode": "windows-msvc-x64",
            "cStandard": "c17",
            "cppStandard": "c++17"
        }
    ],
    "version": 4
}
	\end{lstlisting}
	
	\item[Alternativa B: Creación de un Target ficticio en CMake]~\\
	Agregue al final de su archivo \texttt{CMakeLists.txt} raíz una biblioteca ficticia agrupando todos los archivos de prueba inactivos, pero calificada con la directiva \texttt{EXCLUDE\_FROM\_ALL} para evitar que el compilador los enlace realmente en el archivo binario:
	\begin{lstlisting}[style=CMakeStyle]
# Registrar las fuentes inactivas en CMake solo para que IntelliSense las indexe
add_library(inactive_mains EXCLUDE_FROM_ALL
    Core/Src/main.c
    Core/Src/main_1.c
    Core/Src/main_2.c
)
target_link_libraries(inactive_mains PRIVATE stm32cubemx)
target_include_directories(inactive_mains PRIVATE Core/HDC1080)
	\end{lstlisting}
\end{description}
