\section{Inicialización del backend (C\# .NET 9 Web API)}
Desde la terminal raíz del proyecto, inicializa la API Web de C\# y agrega el driver del puerto serie.

\subsection{Desde la terminal}
Desde la terminal raíz del proyecto \directory{C:/Visual Studio Projects/NucleoBoard\_Led\\Control} inicializa la API web de C\# y agrega el driver del puerto serie utilizando los siguientes comandos:
\begin{lstlisting}[language=DOS]
	Microsoft Windows [Version 10.0.26200.8457]
	(c) Microsoft Corporation. Todos los derechos reservados.
	
	# 1. Muevete a la carpeta Service
	C:\Visual Studio Projects\NucleoBoard_LedControl>cd Service
	
	#2. Crea un nuevo proyecto Web API de C# .NET 9
	C:\Visual Studio Projects\NucleoBoard_LedControl\Service>dotnet new webapi --use-controllers -o LedControllerService
	
	#3. Regresamos a la raiz
	C:\Visual Studio Projects\NucleoBoard_LedControl\Service>cd ..
\end{lstlisting}

El comando de .NET CLI \texttt{dotnet new webapi --use-controllers -o LedController\\Service} crea un nuevo proyecto de \textbf{ASP.NET Core Web API}, el comando se explica a continuación:

\begin{enumerate}
	\item \commandbox{dotnet}: Es la herrramienta de línea de comandos de .NET. Se usa para crear proyectos, compilarlos, ejecutarlos, restarurar paquetes, publicar, etc.
	\item \commandbox{new}: Indica que quieres crear algo nuevo. En este caso, crearás un proyecto basado en una plantilla.
	\item \commandbox{webapi}: Es la plantilla que se va a usar. La plantilla \textbf{webapi} genera un proyecto de \textbf{API REST} con \textbf{ASP.NET Core}.
	\item \commandbox{--use-controllers}: Esta opción le dice a la plantilla que use el enfoque de \textbf{controladores MVC} en lugar de depender solo de \textbf{Minimal APIs}. Con esta opción, el proyecto estará preparado para trabajar con clases como estas:
	\begin{lstlisting}[style=VSCodeDark]
		[ApiController]
		[Route("api/[controller]")]
		public class LedController : ControllerBase
		{
			[HttpGet]
			public IActionResult Get()
			{
				return Ok("LED encendido");
			}
		}\end{lstlisting}
	\item \commandbox{-o LedControllerService} La opción \textbf{-o} significa \textbf{output}. Le indica a \texttt{dotnet} que cree el proyecto dentro de una carpeta llamada \texttt{LedControllerService}. Si la carpeta no existe, la crea automáticamente.
\end{enumerate}



Justo después de crear el proyecto, lo vinculamos a la solución para que el IDE sepa que forman parte del mismo ecosistema:
\begin{lstlisting}[language=DOS]
	Microsoft Windows [Version 10.0.26200.8457]
	(c) Microsoft Corporation. Todos los derechos reservados.
	
	C:\Visual Studio Projects\NucleoBoard_LedControl>dotnet sln add Service/LedControllerService/LedControllerService.csproj
\end{lstlisting}

Una vez que el proyecto está creado y vinculado, entramos a su carpeta para agregar las dependencias de terceros (como el puerto serie).
\begin{lstlisting}[language=DOS]
	Microsoft Windows [Version 10.0.26200.8457]
	(c) Microsoft Corporation. Todos los derechos reservados.
	
	C:\Visual Studio Projects\NucleoBoard_LedControl>cd Service
	C:\Visual Studio Projects\NucleoBoard_LedControl\Service>cd LedControllerService 
	C:\Visual Studio Projects\NucleoBoard_LedControl\Service\LedControllerService>dotnet add package System.IO.Ports --version 10.0.8
\end{lstlisting}

\begin{figure}[H]
	\centering 
	\includegraphics[width=1\textwidth]{vscode_17} 
	\caption{Crear nueva plantilla e instalar paquete Nuget.}\label{fig18} 
\end{figure}



El comando \texttt{dotnet add package System.IO.Ports --version 10.0.8} sirve para agregar una dependencia NuGet a tu proyecto .NET, específicamente el paquete \texttt{System.IO.\\Ports} en la versión \texttt{10.0.8}

\begin{enumerate}
	\item \commandbox{dotnet}: Es la herramienta de línea de comandos de .NET CLI, usada para crear, compilar, ejecutar y administrar proyectos y paquetes.
	\item \commandbox{add package}: Indica que quieres agregar un paquete NuGet al proyecto actual. 
	\item \commandbox{System.IO.Ports}: Es el nombre del paquete NuGet que proporciona clases para trabajar con puertos seriales; el tipo principal que expone es SerialPort. 
	\item \commandbox{--version 10.0.8}: Le indica a la CLI que agregue exactamente esa versión del paquete, en lugar de usar automáticamente la última versión compatible disponible.
\end{enumerate}

Cuando ejecutas ese comando, .NET hace varias cosas automáticamente: \\\\
Primero verifica la compatibilidad entre el paquete y los frameworks del proyecto; si es compatible, agrega (o actualiza) un elemento <PackageReference> en el archivo \texttt{.csproj}; y después ejecuta \commandbox{dotnet restore} para descargar el paquete y generar los archivos de assets necesarios.

En la práctica, tu archivo \texttt{.csproj} terminaría con algo parecido a esto
\begin{lstlisting}[style=VSCodeDark]
<ItemGroup>
	<PackageReference Include="System.IO.Ports" Version="10.0.8" />
</ItemGroup>
\end{lstlisting}
\begin{InfoBox}{Sin \texttt{ - -version}} 
	La herramienta intentará resolver una versión apropiada del paquete, normalmente la más reciente compatible con tu proyecto. La documentación indica que, si el paquete ya existe, el comando también puede actualizar la referencia a una versión compatible.
\end{InfoBox}

\begin{enumerate}
	\item \commandbox{npx} (Node Package Runner): Es una herramienta que viene con Node.js y sirve para ejecutar paquetes sin tener que instalarlos de forma global en tu computadora. En lugar de instalar permanentemente el CLI de Angular (\commandbox{npm install -g @angular/cli}), \commandbox{npx} descarga una versión temporal de la herramienta en la memoria, crea tu proyecto, y luego libera el espacio. Esto mantiene tu sistema limpio.
	\item \commandbox{-y} (Yes): Es un parámetro para omitir preguntas. Le dice a Node: \textit{"Si me vas a preguntar si deseo descargar el paquete de Angular CLI, responde 'Sí' automáticamente"}. Esto hace que el comando se ejecute sin pausas interactivas.
	\item \commandbox{@angular/cli}: Es la librería oficial de la línea de comandos de Angular (Angular CLI). Es la herramienta responsable de estructurar la aplicación, configurar el compilador, y generar los archivos base.
	\item \commandbox{new frontend}: Es el comando interno de Angular CLI para crear un nuevo proyecto. El nombre que se le asignará al proyecto y a la aplicación principal es \textbf{frontend}.
	\item \commandbox{--directory=frontend}: Le indica al instalador dónde crear los archivos. Por defecto, Angular crea una nueva carpeta con el nombre del proyecto. Con \commandbox{--directory=frontend}, le indicamos que coloque todos los archivos directamente dentro de la carpeta frontend, evitando que se genera una estructura redundante \commandbox{frontend/frontend/...}
	\item \commandbox{--style=css}: Define qué motor de estilos utilizará la aplicación. En este caso, le decimos que configure hojas de estilo en CSS tradicional (Vanilla CSS) en lugar de preprocesadores más complejos como SCSS, Sass o Less.
	\item \commandbox{--routing=true}: Configura automáticamente el sistema de rutas de Angular (Angular Router). Crea los archivos de configuración de navegación (como \texttt{app.route.ts}) permitiendo que puedas definir páginas y secciones de URL en el futuro muy fácilmente.
	\item \commandbox{--ssr=false}: Desactiva el \textbf{Server-Side Rendering (SSR)}. Angular moderno intenta activar SSR por defecto (lo que significa que un servidor Node procesa el HTML inicial). Como nosotros estamos creando una arquitectura limpia donde el servidor es de C\# .NET, solo necesitamos que Angular compile como una aplicación de cliente clásica (Single Page Application - SPA), la cual es más rápida de compilar, más ligera y se ejecuta directamente en el navegador del usuario.

\subsection{Crear el Servicio Serial (SerialService)}
El archivo \texttt{SerialService} se encargará de gestionar directamente la comunicación física con la terjeta STM32 Nucleo. Este servicio se registrará como un \textbf{Singleton} (una instancia única en memoria), debido a que el puerto COM físico sólo puede estar abierto por un único proceso a la vez. Este archivo realizará lo siguiente:

\begin{description}
	\item[Abrir y cerrar la conexión] con el puerto COM seleccionado por el usuario y a la velocidad indicada.
	\item[Auto conexión inteligente]: Guardará los datos del último puerto exitoso en un archivo local \texttt{serial-settings.json}. Si al arrancar el servidor detecta que la placa está conectada en ese puerto, se conectará de inmediato.
	\item[Lectura asíncrona (Background Thread)]: Creará un hilo de ejecución secundario (Thread) dedicado únicamente a escuchar datos provenientes de la tarjeta en segundo plano. Esto evita que el servidor Web API se congele mientras espera los datos del puerto serie.
	\item[Notificaciones en tiempo real] Utiliza \textbf{SignalR} (\texttt{IHubContext<SerialHub>}) para "empujar" de forma reactiva los datos recibidos (RX), transmitidos (TX) y el estado del LED hacie el frontend, sin que el cliente tenga que estar preguntando periodicamente (polling).
\end{description}

\subsubsection{Namespaces}
En C\# (y otros lenguajes como C++ o Java), la plabra clave \texttt{using} se utiliza para importar Namespaces.
Los Namespaces pueden visualizarse como carpetas organizadoras llenas de código escrito por ti o por terceros. En lugar de escribir el nombre completo de una clase cada vez (como \texttt{System.IO.Ports.SerialPort}), se coloca \texttt{using} al inicio y se puede escribir simplemente como \texttt{SerialPort}.


\begin{enumerate}
	\item \texttt{\textcolor{blue}{using} \textcolor{purple}{LedControllerService.Hubs;}}
	\begin{description}
		\item[Significado] Importa la carpeta virtual de nuestro propio proyecto donde declararemos el canal del WebSocket de SignalR.
		\item[uso] Permite que este servicio haga referencia al canal \texttt{SerialHub} para poder enviarle mensajes a los navegadores web.
	\end{description}
	\item \texttt{\textcolor{blue}{using} \textcolor{purple}{Microsoft.AspNetCore.SignalR;}}
	\begin{description}
		\item[Significado] Importa herramientas oficiales de Microsoft para la comunicación en tiempo real (SignalR).
		\item[uso] Nos da acceso a la interfaz IHubContext, la cual permite que una clase de C\# común y corriente (como este servicio) tome el control de las conexiones WebSockets activas para enviarles datos.
	\end{description}
	\item \texttt{\textcolor{blue}{using} \textcolor{purple}{System;}}
	\begin{description}
		\item[Significado] El espacio de nombres base y fundamental de todo .NET.
		\item[uso] Da soporte a tipos y objetos básicos de programación.
	\end{description}
	\item \texttt{\textcolor{blue}{using} \textcolor{purple}{System.IO;}}
	\begin{description}
		\item[Significado] Significa \textit{System Input/Output} (Entrada/Salida de archivos del sistema)
		\item[uso] Nos permite interactuaer con el disco duro de la PC para saber si existe el archivo de configuración (\texttt{File.Exists}), leerlo (\texttt{File.ReadAllText}) y escribir en él (\texttt{File.WriteAllText})
	\end{description}
	\item \texttt{\textcolor{blue}{using} \textcolor{purple}{System.IO.Ports;}}
	\begin{description}
		\item[Significado] El paquete que se instaló mediante NuGet para gestionar puertos serie.
		\item[uso] Es el que hace toda la "magia" del hardware. Nos provee la clase \texttt{SerialPort}, que es el controlador de bajo nivel para detectar puertos USB conectados, abrir la comunicación, fijar baudios y transmitir/recibir texto.
	\end{description}
	\item \texttt{\textcolor{blue}{using} \textcolor{purple}{System.Text.Json;}}
	\begin{description}
		\item[Significado] El serializador JSON oficial de Microsoft
		\item[uso] Convierte la información de configuración en texto formateado JSON para guardarla en el archivo de texto \texttt{JsonSerializer.Serialize} y la vuelve a transformar en datos estructurados de C\# cuando arranca la app \texttt{JsonSerializer.Deserilize}.
	\end{description}
	\item \texttt{\textcolor{blue}{using} \textcolor{purple}{System.Threading;}}
	\begin{description}
		\item[Significado] Herramientas para la gestión de Hilos de ejecución (\textit{Threads}).
		\item[uso] Sirve para crear el hilo paralelo que lee la tarjeta continuamente y usar pausas breves (\texttt{Thread.Sleep}) para que el procesador de la PC descance cuando no hay datos entrantes.
	\end{description}
		\item \texttt{\textcolor{blue}{using} \textcolor{purple}{System.Threading.Tasks;}}
	\begin{description}
		\item[Significado] Libraría para programación asíncrona moderna de .NET
		\item[uso] Nos provee la clase \texttt{Task}, que sirve para ejecutar pequeñas tareas en segundo plano sin interrumpir el flujo principal (como cuando le enviamos la pregunta \texttt{?} a la placa medio segundo después de haber conectado.)
	\end{description}
\end{enumerate}

Las siguientes líneas son los cimientos de la Programación Orientada a Objetos (POO) en C\# y definen la identidad de nuestro archivo.
\begin{lstlisting}[style=VSCodeDark]
namespace LedControllerService.Services
{
	public class SerialService
	{
		
	}
}
\end{lstlisting}

\begin{enumerate}
	\item \texttt{\textcolor{blue}{namespace} \textcolor{Emerald}{LedControllerService.Services}} \\
	Un \texttt{namespace} es un contenedor lógico que sirve para organizar las clases y evitar conflictos de nombres (por ejemplo, si se crea una clase llamada \texttt{Logger}) y Microsoft tiene otra igual, los namespaces evitarán que choquen.
	\begin{itemize}
		\item \texttt{LedControllerService.Services} le dice al compilador de C\#: \textit{"Esta clase pertenece al proyecto \texttt{LedControllerService} y está guardada dentro del grupo lógico \texttt{Services}"}
		\item Por convención, los namespaces reflejan la estructura física de tus carpetas en el disco duro.
	\end{itemize}
	\item \texttt{\textcolor{blue}{public}}\\
	Es un \textbf{modificador de acceso}. Define quién tiene permitido ver y utilizar esta clase. Significa que \textbf{cualquier otra clase} del proyecto (como tu controlador \texttt{SerialController} o el archivo de inicio \texttt{Program.cs}) puede instanciar, leer y ejecutar este servicio. Si fuera \texttt{private} (privado) o \texttt{internal} (interno), su acceso estaría muy restringido.
	\item \texttt{\textcolor{blue}{class}}\\
	Una clase es la plantilla o molde a partir del cual creamos objetos en programación. La palabra clave class le dice a C\# que a continuación definiremos una estructura que contendrá datos (variables como el puerto COM actual) y comportamientos (métodos como \texttt{Connect} o \texttt{Disconnect}).
	\item \texttt{\textcolor{Emerald}{SerialService}}\\
	Es simplemente el nombre que le asignamos a esta clase.
	\begin{itemize}
		\item Por convención en C\#, los nombres de las clases se escriben en PascalCase (comienzan con mayúscula y cada nueva palabra empieza con mayúscula) y deben ser descriptivos de su función. En este caso, indica que es un "Servicio" encargado de la comunicación "Serial".
	\end{itemize}
\end{enumerate}

\subsubsection{Declaración de variables}
Estas declaraciones representan las propiedades y estados internos de nuestro servicio. Se les conoce como campos privados (private fields) y definen los datos que el servicio necesita recordar y utilizar para funcionar.

Antes de ver a detalle cada variable , es importante recordar estos tres puntos claves de la sintaxis en C\#:

\begin{itemize}
	\item \texttt{\textcolor{blue}{private}} (privado): Significa que estas variables sólo existen dentro de esta clase. Ninguna clase externa puede verlas ni modificarlas directamente.
	\item \textbf{El guión bajo (\_)}: Es una convención estándar en C\# para identificar rápidamente que una variable es un campo privado de la clase.
	\item \textbf{El signo de interrogación (?)}: Por ejemplo, \texttt{SerialPort?} o \texttt{string?} indica que la variable es \textbf{nuleable} (puede valer \textbf{null}, si, por ejemplo, la tarjeta está desconectada).
\end{itemize}

private int _savedBaudRate = 115200;
private Thread? _readThread;
private bool _isRunning;
private readonly object _lock = new();
private readonly string _settingsPath;
\end{lstlisting}

\begin{enumerate}
	\item \textbf{Comunicación en tiempo real y logs}
	\begin{itemize}
		\item \texttt{\textcolor{blue}{private readonly} \textcolor{Emerald}{IHubContext<SerialHub>} \textcolor{SkyBlue}{\_hubContext};}
		\begin{description}
			\item[Qué es:] La interfaz para interactuar con nuestro hub de SignalR (WebSockets).
			\item[Uso] Es la que nos permite hacer \texttt{\_hubContext.Clients.All.SendAsync(...)} para enviar datos de forma instantánea a los navegadores.
		\end{description}
		
		\item \texttt{\textcolor{blue}{private readonly} \textcolor{Emerald}{ILogger<SerialService>} \textcolor{SkyBlue}{\_logger};}
		\begin{description}
			\item[Qué es:] El grabador de logs del sistema de ASP.NET Core.
			\item[Uso] Escribe mensajes en la consola negra del servidor (p.e. \texttt{\_logger.Log\\Information("Puerto conectado")}) para que pueda monitorear en tiempo real qué hace el backend.
		\end{description}
	\end{itemize}
	\item \textbf{Control de la conexión física}
	\begin{itemize}
		\item \texttt{\textcolor{blue}{private} \textcolor{Emerald}{SerialPort}? \textcolor{SkyBlue}{\_serialPort};}
		\begin{description}
			\item[Qué es:] El objeto físico que controla el puerto COM.
			\item[Uso:] Contiene los métodos nativos del sistema para abrir, cerrar, leer y escribir datos en el cable USB. Vale \textbf{null} cuando no hay conexión.
		\end{description}
		
		\item \texttt{\textcolor{blue}{private} \textcolor{Emerald}{string}? \textcolor{SkyBlue}{\_currentPort};}
		\begin{description}
			\item[Qué es:] El nombre del puerto activo, por ejemplo, \texttt{"COM3"}.
		\end{description}
		
		\item \texttt{\textcolor{blue}{private int} \textcolor{SkyBlue}{\_currentBaudRate} =  \textcolor{Dandelion}{115200};}
		\begin{description}
			\item[Qué es:] La velocidad de comunicación activa del puerto (115200 bits por segundo por defecto).
		\end{description}
	\end{itemize}
	\item \textbf{Recordar la última conexión (Persistencia)}
	\begin{itemize}
		\item \texttt{\textcolor{blue}{private string}? \textcolor{SkyBlue}{\_savedPort};}
		\item \texttt{\textcolor{blue}{private int}? \textcolor{SkyBlue}{\_savedBaudRate} = \textcolor{Dandelion}{115200};}

			\item[Qué es:] Variables temporales que guardan en memoria el último puerto y baudios existosos cargados desde el archivo JSON de configuración.
		\end{description}
		
		\item \texttt{\textcolor{blue}{private readonly string}? \textcolor{SkyBlue}{\_settingsPath};}
		\begin{description}
			\item[Qué es:] La ruta absoluta en el disco duro donde se creará y leerá el archivo de configuración \texttt{serial-settings.json}.
		\end{description}
	\end{itemize} 
	\item \textbf{Hilos y control de concurrencia (Multithreading)}
	\begin{itemize}
		\item \texttt{\textcolor{blue}{private} \textcolor{Emerald}{Thread}? \textcolor{SkyBlue}{\_readThread};}
		\begin{description}
			\item[Qué es:] El objeto que representa nuestro hilo secundario de procesamiento.
			\item[Uso:] Ejecuta el bucle de lectura del puerto de forma aislada para no bloquear la página web principal.
		\end{description}
		
		\item \texttt{\textcolor{blue}{private bool} \textcolor{SkyBlue}{\_isRunning};}
		\begin{description}
			\item[Qué es:] Una bandera booleana (verdadero/falso).
			\item[Uso:] Controla el buble de lectura. Si es \texttt{true}, el hilo sigue leyendo; cuando es \texttt{false} (al desconectar), el hilo se apaga de forma limpia.
		\end{description}
		
		\item \texttt{\textcolor{blue}{private readonly object} \textcolor{SkyBlue}{\_lock} = \textcolor{blue}{new}();}
		\begin{description}
			\item[Qué es:] Un objeto de bloqueo exclusivo.
			\item[Uso:] Como tenemos varios hilos de ejecución activos (el de lectura de datos, y el de la web que envía comandos como encender el LED), este objeto sirve de "semáforo" para asegurar que un sólo hilo a la vez intente comunicarse con la tarjeta STM32, evitando que el puerto colapse por peticiones simultáneas.
		\end{description}
	\end{itemize}
	
\end{enumerate}

\subsubsection{Constructor de la clase}
El constructor es una función especial que se \textbf{ejecuta automáticamente una sola vez} en el momento exacto en que el servidor crea la clase en memoria. Su objetivo es inicializar los datos y preparar la clase para que pueda ser utilizada.

Se le reconoce porque \textbf{tiene el mismo nombre que la clase} y no devuelve ningún valor.
\begin{lstlisting}[style=VSCodeDark]
public SerialService(IHubContext<SerialHub> hubContext, ILogger<SerialService> logger)
{
	_hubContext = hubContext;
	_logger = logger;
	// Ruta del archivo JSON para guardar las configuraciones de conexion
	_settingsPath = Path.Combine(AppDomain.CurrentDomain.BaseDirectory, "serial-settings.json");
	LoadSettings();
	
	// Intenta auto-conectar al iniciar el servidor
	TryAutoConnect();
}
\end{lstlisting}

Los parámetros \texttt{hubContext} y \texttt{logger} se reciben utilizando un patrón de diseño fundamental en ASP.NET Core llamado \textbf{Inyección de Dependencias}.

\begin{description}
	\item[Cómo funciona:] En lugar de que nuestro código intente crear manualmente los servicios de SignalR o de Logs (escribiendo un \texttt{new HubContext(...)} lo cual sería muy complejo), el framework de .NET se encarga de crear esos servicios al arrancar el servidor.
	\item[La inyección:] Al declarar \texttt{IHubContext<SerialHub> hubContext} en los parámetros de entrada del constructor, le estamos diciendo a .NET \textit{"necesito usar SignalR y Logs, por favor pásamelos cuando crees este servicio"}. .NET los busca y los "inyecta" automáticamente en la función.
	\item[El guardado:] Las primeras dos líneas
	\begin{lstlisting}[style=csharpStyle]
		\_hubContext = hubContext;
		\_logger = logger;
	\end{lstlisting}
	Toman esos servicios provistos por .NET y los guardan en nuestras variables privadas (los campos con guión bajo) para que cualquier otra función de la clase pueda usarlos más tarde.
\end{description}

\begin{InfoBox}{El constructor de la clase} 
	A nivel técnico, toda clase tiene siempre un constructor, pero no siempre tienes que escribirlo tú. \\
	
	Si crea una clase y no escribes ningún constructor, el compilador de C\# creará uno automáticamente tras bambalinas. A esto se le conoce como \textbf{Constructor por Defecto} (Default Constructor). Por ejemplo, si escribe esto
	\begin{lstlisting}[style=csharpStyle]
	public class Persona
	{
		public string Nombre { get; set; }
	}
	\end{lstlisting}
	El compilador lo transforma en esto:
	\begin{lstlisting}[style=csharpStyle]
		public class Persona
		{
			public string Nombre { get; set; }
			
			//Creado automaticamente por C#
			public Persona()
			{
			}
		}
	\end{lstlisting}
	Debes escribir manualmente un constructor en tres casos principales:
	\begin{itemize}
		\item \textbf{Para inyección de dependencias (como en nuestro proyecto)}: Si tu clase necesita que .NET le pase herramientas externas al iniciar (como el contexto de SignalR o el Logger), debes escribir el constructor para declarar qué dependencias requieres.
		\item \textbf{Para inicializar datos obligatorios:} Si deseas que no se pueda crear una clase sin ciertos datos.
		\item \textbf{Para ejecutar código de preparación:} Si necesitas realizar acciones preias (como calcular rutas de archivos, conectar una base de datos o suscribir eventos.)
	\end{itemize}
\end{InfoBox}

\textbf{Función del cuerpo de la función} \\

\begin{enumerate}
	\item \textbf{Calcular la ruta física del archivo de configuración}

La siguiente función es sencilla pero fundamental: se encarga de consultar el sistema operativo Windows qué puertos COM físicos o virtuales están activos y utilizables en la computadora en ese instante. 
\begin{lstlisting}[style=csharpStyle]
public string[] GetAvailablePorts()
{
	return SerialPort.GetPortNames();
}
\end{lstlisting}

\begin{itemize}
	\item \texttt{public}:  Permite que clases externas (como nuestro controlador API \texttt{SerialCont\\roller}) llamen a esta función para obtener la lista de puertos.
	\item \texttt{string[]}: Es el tipo de dato que devuelve la función. En este caso, una lista o "vector" de textos (por ejemplo: ["COM1", "COM3"]).
	\item \texttt{SerialPort.GetPortNames()}: Es un método nativo provisto por .NET (\texttt{System\\.IO.Ports}). Este método le pregunta a Windows: \textit{"Oye, ¿qué dispositivos seriales están conectados ahora mismo?"}.
	\item \texttt{return}: Envía la lista de puertos obtenida de vuelta a quien haya llamado a la función.
\end{itemize}

Cuando conectas tu tarjeta de desarrollo STM32 Nucleo mediante cable USB a la computadora, Windows le asigna automáticamente un puerto serie virtual. Esta función permite que, cuando abras la página web del panel de control en tu navegador, el menú desplegable (Select) se llene automáticamente con los puertos reales disponibles en tu PC, evitando que tengas que escribir a mano a qué puerto conectarte. Si no hay nada conectado, devolverá una lista vacía.

\begin{lstlisting}[style=VSCodeDark]
public object GetStatus()
{
	lock (_lock)
	{
		return new
		{
			IsConnected = _serialPort?.IsOpen ?? false,
			PortName = _serialPort?.IsOpen == true ? _currentPort : null,
			BaudRate = _serialPort?.IsOpen == true ? _currentBaudRate : 0,
			SavedPort = _savedPort,
			SavedBaudRate = _savedBaudRate
		};
	}

\end{lstlisting}


Esta función se encarga de recopilar e informar el estado actual de la conexión serie y la configuración del servidor, empaquetándolo todo en un formato que el navegador pueda entender fácilmente.

\begin{enumerate}
	\item \textbf{El tipo de retorno} \texttt{object}\\
	En C\#, \texttt{object} es la clase base de la cual heredan todos los tipos de datos. Retornar un objeto \texttt{object} nos permite crear y devolver un\textbf{ Objeto Anónimo} (un objeto "al vuelo", sin necesidad de crear una clase formal para él).\\
	Cuando una respuesta es enviada al navegador a través de nuestra API, .NET la convierte automáticamente en un formato JSON limpio y estructurado.
	\item \textbf{El semáforo} \texttt{lock (\_lock) {} }\\
	Dado que el puerto serie puede estar siendo modificado por varios procesos al mismo tiempo (por ejemplo, el hilo de lectura secundario o una orden de desconexión repentina), la instrucción lock (\texttt{\_lock}) detiene temporalmente cualquier otra operación sobre el puerto mientras recopilamos el estado. Esto evita errores de concurrencia y caídas del servidor.
	\item \textbf{Las propiedades que devuelve (lógica paso a paso)}\\
	\begin{itemize}
		\item \texttt{IsConnected = \_serialPort?.IsOpen ?? false}
		\begin{itemize}
			\item \textit{Lógica}: Usa dos operadores modernos de C\#:
			\begin{itemize}
				\item El operador nulo-condicional (?): Si \texttt{SerialPort} es nulo (no existe), devuelve nulo en lugar de lanzar un error.
				\item El operador de coalescencia nula (??): Si la parte izquierda es nula, toma el valor de la derecha, \texttt{false}. 
			\end{itemize}
			\item \textit{Resultado}: Devuelve \texttt{true} si el puerto está físicamente conectado y abierto, o \texttt{false} si no lo está.
		\end{itemize}
		\item \texttt{PortName = \_serialPort?.IsOpen == true ? \_currentPort : null}\\
		\begin{itemize}
			\item \textit{Lógica}: Es un operador ternario \texttt{(condición ? si\_es\_verdadero : si\_es\_falso)}.
			\item \textit{Resultado}: Si el puerto está abierto, devuelve el nombre del puerto activo (ej: "COM3"). Si está cerrado, devuelve \texttt{null}.
		\end{itemize}
		\item \texttt{BaudRate = \_serialPort?.IsOpen == true ? \_currentBaudRate : 0}
		\begin{itemize}
			\item \textit{Resultado}: Si está conectado, devuelve la velocidad actual (ej: 115200). Si no, devuelve 0.
		\end{itemize}
		\item \texttt{SavedPort = \_savedPort} y \texttt{SavedBaudRate = \_savedBaudRate}
		\begin{itemize}
			\item \textit{Resultado}: Devuelve la última configuración de puerto y baudios que se guardó en el archivo JSON. Esto sirve para que el frontend sepa qué puerto autoseleccionar por defecto en la pantalla al cargar la página.
		\end{itemize}
	\end{itemize}
\end{enumerate}

\newpage
\textbf{\large ¿Qué es un Tipo Anónimo?}\\\\
Cuando se utiliza la palabra clave \texttt{new} seguida de llaves \texttt{\{...\} } sin especificar el nombre de una clase, le estás diciendo a C\#: \textit{"por favor crea una clase temporal en segundo plano, asígnale propiedades con estos nombres, y deduce sus tipos de datos según los valores que les estoy asignando a la derecha"}.

\begin{InfoBox}{Tipos Anónimos} 
En este fragmento de código:

\begin{lstlisting}[style=VSCodeDark]
return new
{
	IsConnected = _serialPort?.IsOpen ?? false, // C# deduce que es un booleano (bool)
	PortName = _serialPort?.IsOpen == true ? _currentPort : null, // C# deduce que es un texto (string)
	BaudRate = _serialPort?.IsOpen == true ? _currentBaudRate : 0, // C# deduce que es un entero (int)
	SavedPort = _savedPort,
	SavedBaudRate = _savedBaudRate
};
\end{lstlisting}
El compilador crea en tiempo de compilación una clase temporal oculta que contiene exactamente este aspecto:

\begin{lstlisting}[style=VSCodeDark]
public class ClaseAnonimaOculta
{
	public bool IsConnected { get; }
	public string PortName { get; }
	public int BaudRate { get; }
	public string SavedPort { get; }
	public int SavedBaudRate { get; }
}
\end{lstlisting}
\end{InfoBox}

\textbf{\large ¿Por qué hacemos esto en lugar de crear una nueva clase?}
\begin{description}
	\item[Ahorro de código]: Nos evita la molestia de tener que crear un archivo o clase llamada \texttt{EstadoSerial.cs} que sólo se usaría una vez en todo el proyecto para esta respuesta.
	\item[Compatibilidad con JSON]: El servidor de ASP.NET Core lee las propiedades de este objeto anónimo y las convierte automáticamente en un formato JSON estándar que viaja hacia Angular:
	\begin{lstlisting}[language=json]
	{
		"isConnected": true,
		"portName": "COM3",
		"baudRate": 115200,
		"savedPort": "COM3",
		"savedBaudRate": 115200
	}
	\end{lstlisting}
	Es una característica muy utilizada en .NET para construir respuestas rápidas de APIs.
\end{description}

// Abre la conexion con el puerto serie fisico
public bool Connect(string portName, int baudRate, out string errorMessage)
{
	lock (_lock)
	{
		errorMessage = string.Empty;
		if (_serialPort != null && _serialPort.IsOpen)
		{
			Disconnect();
		}
		try
		{
			_serialPort = new SerialPort(portName, baudRate)
			{
				ReadTimeout = 1000,
				WriteTimeout = 1000,
				NewLine = "\n" // Sincroniza las lecturas por caracteres de salto de linea
			};
			_serialPort.Open();
			// Limpiar buffers iniciales para descartar basura de conexion
			_serialPort.DiscardInBuffer();
			_serialPort.DiscardOutBuffer();
			_currentPort = portName;
			_currentBaudRate = baudRate;
			_savedPort = portName;
			_savedBaudRate = baudRate;
			
			SaveSettings(portName, baudRate);
			// Inicializa el hilo de lectura secundario
			_isRunning = true;
			_readThread = new Thread(ReadLoop)
			{
				IsBackground = true,
				Name = "SerialReadThread"
			};
			_readThread.Start();
			_logger.LogInformation($"Puerto serie {portName} abierto con exito a {baudRate} bps.");
			
			// Notificar a los clientes web a traves de SignalR
			_hubContext.Clients.All.SendAsync("ReceiveStatus", new { IsConnected = true, PortName = portName, BaudRate = baudRate });
			_hubContext.Clients.All.SendAsync("ReceiveTxLog", $"[Sistema] Puerto COM conectado con exito a {baudRate} bps.");
			
			// Preguntar proactivamente el estado del LED a la placa ('?') tras conectar
			Task.Run(async () =>
			{
				await Task.Delay(500);
				SendCommand("?");
			});
			return true;
		}
		catch (Exception ex)
		{
			errorMessage = ex.Message;
			_logger.LogError($"Error al abrir el puerto {portName}: {ex.Message}");
			_hubContext.Clients.All.SendAsync("ReceiveTxLog", $"[Sistema] Error de conexion en {portName}: {ex.Message}");
			return false;
		}
	}
}
\end{lstlisting}

Esta función se encarga de realizar la desconexión controlada y la liberación de recursos del hardware en tu computadora. // Cierra la conexion y libera los recursos del puerto
public void Disconnect()
{
	lock (_lock)
	{
		_isRunning = false;
		
		if (_serialPort != null)
		{
			try

				if (_serialPort.IsOpen)

					_serialPort.Close();

			}

				_serialPort.Dispose();

		}
		_currentPort = null;
		_logger.LogInformation("Puerto serie desconectado.");
		_hubContext.Clients.All.SendAsync("ReceiveStatus", new { IsConnected = false, PortName = (string?)null, BaudRate = 0 });
		_hubContext.Clients.All.SendAsync("ReceiveTxLog", "[Sistema] Puerto serie desconectado.");
	}

	{

							SendCommand("?");
						});

			private void TryAutoConnect()
			{

			// Carga configuraciones de serial-settings.json
			private void LoadSettings()
			{
				try
				{
					if (File.Exists(_settingsPath))
					{
						string json = File.ReadAllText(_settingsPath);
						var settings = JsonSerializer.Deserialize<SettingsData>(json);
						if (settings != null)
						{
							_savedPort = settings.SavedPort;
							_savedBaudRate = settings.SavedBaudRate;
						}
					}
				}
				catch (Exception ex)
				{
					_logger.LogError($"No se pudo cargar la configuracion de puerto: {ex.Message}");
				}
			}
			// Guarda configuraciones en serial-settings.json
			private void SaveSettings(string portName, int baudRate)
			{
				try
				{
					var settings = new SettingsData { SavedPort = portName, SavedBaudRate = baudRate };
					string json = JsonSerializer.Serialize(settings, new JsonSerializerOptions { WriteIndented = true });
					File.WriteAllText(_settingsPath, json);
				}
				catch (Exception ex)
				{
					_logger.LogError($"No se pudo guardar la configuracion de puerto: {ex.Message}");
				}
			}
			private class SettingsData
			{
				public string? SavedPort { get; set; }
				public int SavedBaudRate { get; set; }
			}
		}
	}
\end{lstlisting}

\subsection{Crear el Hub de SignalR (\texttt{SerialHub.cs})}
Un \textbf{Hub} (Concentrador) en SignalR es el canal lógico de comunicación donde se conectan los clientes externos (los navegadores web) a través de \textbf{WebSockets}. Funciona como un conmutador electrónico.\\

\begin{enumerate}
	\item Dentro de Visual Studio Code, dirígete a la carpeta \directory{Service/LedControllerService}
	\item Crea una nueva carpeta llamada \textbf{Hubs}
	\item Dentro de la carpeta \textbf{Hubs}, crea un archivo llamado \texttt{SerialHub.cs}
	\item Pega el siguiente código en el archivo
\end{enumerate}

\begin{lstlisting}[style=VSCodeDark]
using Microsoft.AspNetCore.SignalR;
using System.Threading.Tasks;

namespace LedControllerService.Hubs
{
	public class SerialHub : Hub
	{
		// Este Hub puede permanecer vacio ya que toda la emision de datos 
		// la controla el servicio singleton 'SerialService' utilizando IHubContext<SerialHub>.
		// Sin embargo, permite que los clientes se conecten mediante WebSockets directamente.
		
		public override async Task OnConnectedAsync()
		{
			// Este metodo se dispara automaticamente cuando un navegador web se conecta.
			// Es util si quisieramos registrar informacion del cliente conectado en el futuro.
			await base.OnConnectedAsync();
		}
	}
}
\end{lstlisting}

\subsubsection{¿Qué significan los componentes de este código?}
\begin{itemize}
	\item \texttt{using Microsoft.AspNetCore.SignalR;}: Importa la librería oficial de SignalR que provee la clase \texttt{Hub}.
	\item \texttt{namespace LedControllerService.Hubs}: Define que la clase se encuentra en la carpeta lógica Hubs de tu proyecto.
	\item \texttt{public class SerialHub : Hub}: La palabra clave \texttt{:} significa \textbf{herencia}. Le estamos indicando a C\# que nuestra clase \texttt{SerialHub} hereda todas las propiedades y comportamientos de la clase base \texttt{Hub} de Microsoft, convirtiéndose oficialmente en un canal de WebSockets de SignalR.
	\item \texttt{public override async Task OnConnectedAsync()}:
	\begin{itemize}
		\item \texttt{override} (Sobrescribir): Indica que estamos reemplazando el comportamiento por defecto de lo que hace SignalR cuando alguien se conecta.
		\item \texttt{async Task}: Define que es un método asíncrono (que no congela la ejecución mientras espera una respuesta de red).
	\end{itemize} 
\end{itemize}

\begin{InfoBox}{Advertencias después de guardar el archivo SerialHub.cs } 
	Al guardar este archivo, la advertencia de \texttt{using LedControllerService.Hubs;} en el archivo \texttt{SerialService.cs} desaparecerán.
\end{InfoBox}

\subsection{Crear el controlador API (\texttt{SerialController.cs})}

El controlador es la puerta de entrada para las peticiones HTTP convencionales (REST) del frontend. Mientras que SignalR maneja la recepción continua de datos en tiempo real, el controlador maneja las órdenes directas que realiza el usuario mediante clics en la interfaz (como \textit{"conéctame a este puerto"}, \textit{"desconéctame"} o \textit{"envía este comando al LED"}).

\begin{description}
	\item[Inyección de dependencias] El controlador no tiene idea de cómo hablar con el puerto serie. En su lugar, el constructor recibe nuestro \texttt{SerialService} y le delega todo el trabajo.
	\item[Expoints expuestos] listado de endpoints
	\begin{enumerate}
		\item \texttt{GET /api/serial/ports}: Obtiene la lista de puertos COM
		\item \texttt{GET /api/serial/status}: Obtiene el estado actual de la conexión.
		\item \texttt{POST /api/serial/connect}: Ordena la conexión a un puerto y baudios específicos.
		\item \texttt{POST /api/serial/disconnect}: Cierra la conexión activa.
		\item \texttt{POST /api/serial/send}:Envía un comando a la tarjeta (\texttt{0}, \texttt{1}, \texttt{?}).
	\end{enumerate}
\end{description}

\begin{enumerate}
	\item En tu editor, ve a la carpeta que ya existe en tu proyecto: \texttt{Service/LedController\\Service/Controllers/}
	\item Dentro de esa carpeta, crea un archivo llamado \texttt{SerialController.cs}.
	\item Pega el siguiente código en el archivo:
\end{enumerate}

\begin{lstlisting}[style=VSCodeDark]
using LedControllerService.Services;
using Microsoft.AspNetCore.Mvc;

namespace LedControllerService.Controllers
{
	[ApiController]
	[Route("api/[controller]")] // Define la ruta base como: api/serial
	public class SerialController : ControllerBase
	{
		private readonly SerialService _serialService;
		
		// El constructor recibe el servicio singleton SerialService por Inyeccion de Dependencias
		public SerialController(SerialService serialService)
		{
			_serialService = serialService;
		}
		
		// GET: api/serial/ports
		[HttpGet("ports")]
		public ActionResult<string[]> GetAvailablePorts()
		{
			return Ok(_serialService.GetAvailablePorts());
		}
		
		// GET: api/serial/status
		[HttpGet("status")]
		public ActionResult GetStatus()
		{
			return Ok(_serialService.GetStatus());
		}
		
		// POST: api/serial/connect
		[HttpPost("connect")]
		public ActionResult Connect([FromBody] ConnectRequest request)
		{
			if (string.IsNullOrEmpty(request.PortName))
			{
				return BadRequest("El nombre del puerto es obligatorio.");
			}
			
			if (request.BaudRate <= 0)
			{
				return BadRequest("La velocidad de baudios debe ser mayor a cero.");
			}
			
			string error;
			bool success = _serialService.Connect(request.PortName, request.BaudRate, out error);
			
			if (success)
			{
				return Ok(new { Message = "Conectado exitosamente.", Success = true });
			}
			else
			{
				return StatusCode(500, new { Message = $"Error de conexion: {error}", Success = false });
			}
		}
		
		// POST: api/serial/disconnect
		[HttpPost("disconnect")]
		public ActionResult Disconnect()
		{
			_serialService.Disconnect();
			return Ok(new { Message = "Desconectado exitosamente.", Success = true });
		}
		
		// POST: api/serial/send
		[HttpPost("send")]
		public ActionResult SendCommand([FromBody] SendCommandRequest request)
		{
			if (string.IsNullOrEmpty(request.Command))
			{
				return BadRequest("El comando no puede estar vacio.");
			}
			
			bool success = _serialService.SendCommand(request.Command);
			
			if (success)
			{
				return Ok(new { Message = "Comando enviado.", Success = true });
			}
			else
			{
				return BadRequest(new { Message = "No se pudo enviar el comando. El puerto esta conectado?", Success = false });
			}
		}
	}
	
	// Modelos de datos para recibir los parametros del cuerpo de las peticiones JSON (POST)
	public class ConnectRequest
	{
		public string PortName { get; set; } = string.Empty;
		public int BaudRate { get; set; } = 115200;
	}
	
	public class SendCommandRequest
	{
		public string Command { get; set; } = string.Empty;
	}
}
\end{lstlisting}

\textbf{Detallles del código}\\
\begin{itemize}
	\item \texttt{[ApiController]}: Le dice a .NET que esta clase responderá peticiones de red HTTP y aplicará validaciones automáticas a los modelos de datos de entrada.
	\item \texttt{[Route("api/[controller]")]}: Configura la URL. La palabra \texttt{[controller]} se reemplaza automáticamente por el nombre de la clase sin la palabra "Controller", por lo que todas las peticiones a esta API comenzarán con \texttt{/api/serial}.
	\item \texttt{[FromBody]}: Indica al compilador que el contenido de la petición (el JSON con el puerto o comando) viene en el cuerpo de la solicitud HTTP (Body) y debe ser transformado automáticamente a las clases \texttt{ConnectRequest} o \texttt{SendCommandRequest}.
\end{itemize}

\subsection{Configuración del archivo \texttt{Program.cs}}
El archivo\texttt{ Program.cs} es el punto de entrada principal del servidor. Cuando inicias tu API con dotnet run, es este archivo el que se ejecuta primero.

Para que todo nuestro sistema funcione, debemos configurarle los siguientes servicios básicos al constructor del servidor:

\begin{description}
	\item[Configurar la política de CORS:] Por seguridad, los navegadores bloquean llamadas entre diferentes puertos (Angular corre en el puerto 4200 y C\# corre en el puerto 5200). Aquí le daremos permiso explícito a \texttt{http://localhost:4200} para realizar peticiones y conectarse por WebSockets (usando \texttt{.AllowCredentials()} )
	\item[Intectar el] \texttt{SerialService}: Registraremos este servicio para que esté siempre activo y disponible para el controlador.
	\item[Activar Signalr]: Habilita el motor de WebSockets en el backend.
	\item[Mapear el Hub de websockets:] Configura la dirección URL física (\texttt{/hubs/serial}) que escuchará las conexiones de SignalR.
	\item[Forzar el puerto del servidor:] Obligaremos a la aplicación a correr exactamente en \texttt{http:\\//localhost:5200} para que coincida con la configuración de llamadas de Angular.
\end{description}

\begin{enumerate}
	\item En el editor, abre el archivo existente \texttt{Service/LedControllerService/Program.cs}
	\item Borra todo su contenido original y reemplázalo por el siguiente código estructurado:
\end{enumerate}

\begin{lstlisting}[style=VSCodeDark]
using LedControllerService.Hubs;
using LedControllerService.Services;

var builder = WebApplication.CreateBuilder(args);

// 1. Configurar CORS (Cross-Origin Resource Sharing) para Angular
builder.Services.AddCors(options =>
{
	options.AddPolicy("CorsPolicy", policy =>
	{
		policy.WithOrigins("http://localhost:4200") // Permite el origen del frontend
		.AllowAnyHeader()
		.AllowAnyMethod()
		.AllowCredentials(); // Obligatorio para permitir conexiones de WebSockets de SignalR
	});
});

// 2. Agregar el soporte para los controladores API REST tradicionales
builder.Services.AddControllers();

// 3. Agregar el soporte para SignalR (WebSockets)
builder.Services.AddSignalR();

// 4. Registrar nuestro SerialService como Singleton (instancia unica)
builder.Services.AddSingleton<SerialService>();

// 5. Agregar OpenAPI (Swagger) para documentación y pruebas
builder.Services.AddOpenApi();

var app = builder.Build();

// Configurar el pipeline de peticiones HTTP
if (app.Environment.IsDevelopment())
{
	app.MapOpenApi();
}

// 6. Activar la política de CORS antes del ruteo de controladores y hubs
app.UseCors("CorsPolicy");

app.UseAuthorization();

// 7. Mapear los Controladores de la API REST
app.MapControllers();

// 8. Mapear la ruta URL para el Hub de SignalR
app.MapHub<SerialHub>("/hubs/serial");

// 9. Forzar al servidor a escuchar en el puerto local 5200
app.Run("http://localhost:5200");

\end{lstlisting}

\begin{itemize}
	\item \texttt{builder.Services.AddSingleton<SerialService>();}: Le dice a .NET: \textit{"Crea una única instancia de \texttt{SerialService} al arrancar. Cuando un controlador (\texttt{Serial\\Controller}) la pida en su constructor, dale siempre esta misma instancia"} (evita que se intente abrir el puerto USB varias veces).
	\item \texttt{app.UseCors("CorsPolicy");}: Activa el semáforo de seguridad de red en el servidor. Debe ir colocado antes de mapear controladores o hubs para que las peticiones web no sean rechazadas.
	\item \texttt{app.MapHub<SerialHub>("/hubs/serial");}: Mapea la dirección de WebSockets. El cliente de Angular intentará conectarse a \texttt{http://localhost:5200/hubs/serial}.
\end{itemize}
